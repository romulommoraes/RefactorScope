# Documentation: Statistics Module (RefactorScope.Statistics)

## Overview

Despite its nomenclature, the RefactorScope.Statistics module acts primarily as an observational engine and payload integrity verifier.

Its main responsibility in the RefactorScope ecosystem is not to perform formal statistical inferences or advanced probabilistic analyses, but rather to attest to the sanity of the reported data and measure how complete the set of evidence generated by the parsers is. The module operates in a strictly non-blocking (fail-safe) manner, ensuring that failures in metrics consolidation do not abort the main execution.

## What the Module Actually Evaluates (Coverage vs. Confidence)

The main indicator derived from this module for the executive dashboards is the Statistics Coverage. This indicator answers the question: "How much of the expected payload arrived usable for executive reading?".

## The coverage calculation evaluates:

Report Availability: Verifies if the StatisticsReport structure was instantiated and returned successfully.

Presence of Fundamental Blocks: Confirms the existence of the confidence structural blocks (ParsingConfidence) and the metrics summary (MetricsStatisticsSummary).

Key Field Population: Validates if primary metrics contain populated and logical values, such as the averages of classes per file, references per class, and the proportion of unresolved candidates (unresolved ratio).

Basic Mathematical Sanity: The engine applies native protections against calculation anomalies, such as preventing division by zero when stipulating file and type totals.

## Limitations and Expectation Alignment

To maintain clarity about the tool's architectural limits, it is documented that the current module DOES NOT measure:

Formal statistical rigor or inferential significance of the data.

Effective sample size or the mathematical confidence of the parsing model.

Advanced methodological consistency or cross-structural stability across multiple continuous executions.

Note: We display the metric on the dashboard as "Coverage" precisely because the module does not yet expose a formal "Evidence Score".

## Architecture and Main Components

The payload integrity flow is divided into three distinct components, separating the calculation engine from orchestration and display:

ValidationEngine: The static and observational engine. Responsible for reading the consolidated structural model, applying mathematical protections, and extracting the StatisticsReport safely (RunSafely), delegating error logs without breaking the pipeline.

StatisticsValidationAnalyzer: Acts as the Adapter pattern. It wraps the ValidationEngine to fit the IAnalyzer contract, allowing the RefactorScope orchestrator to invoke it in a standardized way, respecting configuration opt-in/opt-out rules.

DashboardMetricsCalculator: Located in the export layer (Exporters.Projections), it acts as the bridge between raw data and the interface. It does not generate analyses but "chews" the StatisticsResult to calculate the executive coverage score (CalculateStatisticsCoverageScore) that will feed the HTML reports.
